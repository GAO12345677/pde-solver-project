# 自动优化方向与边界说明

## 1. 先回答你最核心的问题

### 1.1 GNN / PINN 能不能加

能加，但建议顺序是：

1. 先加 `GNN` 算法选择器
2. 再做 `PINN` 原型

原因：

- 当前项目已经有 `RF / XGB / RL / MLP` 算法选择层
- `GNN` 是对“算法选择器”的自然增强
- `PINN` 属于“直接用神经网络求 PDE 解”的路线，改动会大很多
- 课程项目里，先补 `GNN` 更稳、更容易讲清楚，也更容易逐步验证

## 2. 推荐的唯一主优化方向

### 方向：`GNN 算法选择增强`

目标：

- 在现有 `RF / XGB / RL / MLP` 之外，新增一个 `GNN` 策略
- 输入仍然围绕 `physics + hardware + domain`
- 输出仍然是 `fdm / fem / spectral`
- 保持和当前 API / 前端交互方式一致

这样做的优点：

- 不推翻当前架构
- 能和现在的 benchmark 体系直接比较
- 方便老师理解“AI 模块从传统机器学习扩展到了图神经网络”

## 3. 和你想象中的“自动持续优化”有什么边界

### 3.1 可以自动做的

- 重训练模型
- 跑 benchmark
- 跑测试
- 生成状态报告
- 排出下一步最优先任务
- 连续执行这些确定性的任务

### 3.2 不能只靠本地脚本自动做的

- 像我现在这样自主理解代码并写出新代码
- 连续无人值守地修改架构、补新模块、修复杂 bug

原因：

- 这类“自动写代码”需要一个外部 AI 代理或编排器
- 普通仓库脚本本身不会自动变成 Codex / GPT 这样的编码代理

### 3.3 这意味着什么

我可以给你做一个：

- `任务总纲`
- `小步推进清单`
- `自动编排脚本`

它会持续执行一切“能确定自动完成”的工作，并在每轮结束时明确告诉你：

- 已完成什么
- 当前指标怎样
- 下一步最值得写哪段代码

这已经足够适合你后续“点一下开始，过一会儿直接看结果”。

## 4. GNN 方向的拆解总纲

### Phase 0：基线固化

- 固定现有 benchmark
- 固定现有回归测试
- 输出 `RF / MLP / RL` 基线结果

### Phase 1：图数据适配

- 设计 graph schema
- 把 `physics / hardware / domain` 编成图结构
- 提供图数据构建器

### Phase 2：GNN 选择器

- 新增 `gnn_selector.py`
- 提供 `train_gnn()` / `predict_gnn()` / `save_gnn()` / `load_gnn()`
- 初版使用简单 GCN / GraphSAGE 即可

### Phase 3：接口接入

- `/select_algorithm` 支持 `gnn_selector`
- 前端支持选择 `GNN`
- 自动求解链路可切换为 GNN 策略

### Phase 4：实验对比

- 对比 `RF / MLP / RL / GNN`
- 输出准确率、稳定性、耗时
- 写入 benchmark 报告

### Phase 5：再考虑 PINN

- 在 GNN 路线稳定之后，再做一维热方程或波动方程的 PINN 原型
- 先作为“研究扩展模块”，不要一上来改主流程

## 5. 实际最推荐的执行顺序

1. 跑自动编排脚本
2. 看最新报告
3. 按报告里“下一步最优先代码任务”继续开发
4. 再重新运行脚本

## 6. 你现在最该记住的一句话

最适合你当前项目的升级路线，不是立刻做完整 PINN，而是：

`先把 GNN 做成第五个算法选择策略，再用现有 benchmark 体系把它和 RF / MLP / RL 比出来。`
